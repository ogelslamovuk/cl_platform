# CODEX_TASK_ITERATION_04.md

## Задача

Реализовать Iteration 4 v2: допуск и контроль билетных операторов / реселлеров поверх существующего sales/reporting flow.

## Главная цель

Показать, что платформа контролирует не только организаторов и заявки, но и билетную инфраструктуру рынка.

Билетный оператор / реселлер не может просто продавать билеты на любом сайте. Он должен:

- пройти допуск в платформе;
- иметь статус допуска;
- иметь соглашение с платформой;
- иметь тип подключения;
- быть выбран организатором как авторизованный канал продаж;
- продавать через уже существующий контролируемый Channel/Seller flow;
- попадать в уже существующую отчётность.

## Ключевое ограничение

В проекте уже есть mock-продажи, mock-возвраты и отчётность по каналам/реселлерам.

Codex обязан сначала найти и переиспользовать существующие механики, а не создавать новые.

Ожидаемые существующие элементы, которые нужно найти и не дублировать:

- ChannelView;
- sellTicketsByReseller;
- refundTicketByReseller;
- AdminReports;
- существующие sales channels;
- существующие reseller/channel records;
- существующий flow mock-продажи;
- существующий flow mock-возврата;
- существующая отчётность по каналам/реселлерам.

Если названия в коде отличаются, найти фактические эквиваленты поиском по repo.

## Что НЕ делать

Запрещено:

- создавать новый sales flow;
- создавать новый refund flow;
- создавать новую отчётность, если текущая уже покрывает продажи/возвраты/каналы;
- создавать новый Channel/Seller;
- создавать новую сущность ResellerEngine/SellerEngine;
- делать полноценный onboarding wizard реселлера;
- делать реальные API/webhooks;
- делать settlement;
- делать реальные комиссии/взаиморасчёты;
- делать настоящий договорный документооборот;
- делать реальные продажи;
- ломать существующие продажи/возвраты/отчётность.

## 1. Центр Управления: реестр билетных операторов

Добавить или усилить существующий admin/control center блок так, чтобы в Центре Управления был виден реестр билетных операторов / реселлеров.

Название блока/раздела использовать на русском, например:

- “Билетные операторы”
- “Реестр билетных операторов”
- “Контроль каналов продаж”

Реестр должен показывать mock-операторов.

Минимальный набор операторов:

- GO2 / собственный виджет платформы
- Национальный билетный оператор
- Городской билетный партнёр
- Региональный реселлер

Названия можно адаптировать под текущий стиль проекта, но все видимые тексты должны быть русскими.

Если в проекте уже есть reseller records, расширить их, а не создавать параллельный список.

## 2. Статусы допуска операторов

У каждого оператора должен быть статус допуска.

Использовать только эти статусы:

- На рассмотрении
- Авторизован
- Требует доработки
- Приостановлен

Правило:

- только “Авторизован” может быть выбран организатором как канал продаж;
- “На рассмотрении”, “Требует доработки”, “Приостановлен” видны как контролируемые, но недоступные для выбора в заявке.

## 3. Тип подключения

У оператора должен быть тип подключения.

Использовать типы:

- API
- Виджет
- Партнёрский канал
- Собственная касса/сайт организатора

Тип подключения должен быть виден:

- в реестре операторов;
- в деталях оператора;
- в заявке организатора на этапе “Каналы продаж”;
- в read-only просмотре заявки в Центре Управления.

## 4. Соглашение с платформой

У оператора должен быть mock-статус соглашения.

Использовать статусы:

- Соглашение подписано
- Ожидает подписания
- Требует обновления

Показать:

- номер/код mock-соглашения;
- дату допуска или дату заявки;
- ответственный контакт;
- статус соглашения.

Не делать настоящий договорный документооборот.
Не генерировать реальные документы.
Не добавлять загрузку реальных договоров.

## 5. Mock-заявки билетных операторов

В Центре Управления показать mock-заявки на подключение операторов.

Это не полноценный onboarding flow.

Нужно только показать, что государственный контур управляет допуском реселлеров.

Минимально показать:

- название оператора;
- дата заявки;
- тип подключения;
- статус заявки;
- контакт;
- комментарий/причина, если “Требует доработки”.

Допустимые mock-действия:

- “Одобрить”
- “Вернуть на доработку”
- “Приостановить”

Если действия уже сложно связать с реестром, достаточно mock-изменения статуса в UI/state.

В результате должно быть видно, что Центр Управления контролирует допуск операторов.

## 6. Предупреждение о запрете неавторизованных продаж

В Центре Управления и на этапе “Каналы продаж” у организатора добавить явное пояснение:

```text
Продажа билетов допускается только через авторизованные каналы, связанные с платформой соглашением.
```

Допустима близкая формулировка, но смысл должен сохраниться.

## 7. Заявка организатора: этап “Каналы продаж”

Усилить этап “Каналы продаж” в wizard заявки организатора.

Должно быть:

- список доступных каналов/операторов;
- статус допуска оператора;
- тип подключения;
- статус соглашения;
- возможность выбрать оператора для события;
- недоступные операторы визуально видны, но выбрать их нельзя;
- подсказка, почему недоступный оператор нельзя выбрать;
- summary выбранных каналов.

Выбор разрешён только для операторов со статусом “Авторизован”.

Операторы “На рассмотрении”, “Требует доработки”, “Приостановлен” должны быть disabled/недоступны для выбора.

## 8. Центр Управления: read-only в заявке

В просмотре поданной заявки Центр Управления должен видеть:

- выбранные каналы продаж;
- выбранных операторов;
- статусы допуска операторов;
- типы подключения;
- статусы соглашения;
- mock-статус интеграции;
- предупреждение, если канал не авторизован или отключён.

Админ не должен вручную настраивать продажи внутри заявки.

Админ только видит и контролирует выбранную конфигурацию каналов.

## 9. Связь с существующим Channel/Seller

После approval события выбранный авторизованный оператор должен попадать в уже существующий Channel/Seller flow.

Важно:

- не создавать новую продажу;
- не создавать новый возврат;
- не создавать новый отчёт;
- использовать существующий Channel/Seller;
- использовать существующие mock sale/refund mechanisms;
- показать, что доступ к продаже открыт для выбранного авторизованного оператора.

Минимально показать:

- карточку доступного события;
- статус “Доступ к продаже открыт”;
- канал/оператор;
- зал/тарифную конфигурацию read-only, если она уже доступна из Iteration 3;
- mock-индикатор интеграции: “Интеграция активна” или “Доступ получен”.

Если в текущем Channel/Seller уже есть mock-продажа и mock-возврат, не менять их глубоко. Только убедиться, что они работают с выбранным авторизованным оператором.

## 10. Связь с существующей отчётностью

Не создавать новую отчётность по каналам/реселлерам, если существующая AdminReports уже показывает продажи/возвраты/каналы/реселлеров.

Нужно обеспечить, чтобы в существующей отчётности было понятно:

- продажи идут через авторизованного оператора;
- оператор связан с платформенным допуском;
- канал продаж контролируется платформой;
- продажи/возвраты остаются в существующих отчётах.

Если в существующей отчётности уже всё видно — не трогать.

Если не видно статуса допуска оператора — аккуратно добавить read-only badge/field в существующий отчёт, не создавая новый отчёт.

## 11. Сквозность

Связь должна быть видна так:

```text
Центр Управления контролирует операторов
-> Организатор выбирает только авторизованные каналы
-> Центр Управления видит выбранные каналы в заявке
-> После approval выбранный оператор видит событие в существующем Channel/Seller
-> Существующие продажи/возвраты проходят через этого оператора
-> Существующая отчётность показывает операции по каналу/реселлеру
```

Это demo-flow, не production-интеграция.
