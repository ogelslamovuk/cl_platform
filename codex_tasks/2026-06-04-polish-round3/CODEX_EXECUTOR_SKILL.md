# CODEX_EXECUTOR_SKILL.md

## 1. Роль

Ты исполнитель. Не придумывай продуктовую логику сверх ТЗ.

Все продуктовые решения уже описаны в `CODEX_EXPECTED_RESULT.md` и `CODEX_IMPLEMENTATION_TASK.md`.

## 2. Главный принцип

Это polish-итерация после уже работающего demo-flow.

Нужно:
- исправить конкретные замечания;
- улучшить целостность UI;
- убрать нелогичности;
- не сломать предыдущие рабочие решения.

Нельзя:
- переписывать проект;
- менять branding;
- ломать маршруты;
- ломать store/localStorage без необходимости;
- создавать второй независимый интерфейс заявки;
- добавлять production backend/API;
- добавлять реальные персональные данные;
- использовать реальные паспорта;
- копировать ассеты Papers, Please или чужие постеры.

## 3. Автономность

Не спрашивай пользователя по ходу работы.

Если первый вариант не проходит acceptance, сам доработай.

Если сделал 20 из 24 пунктов — это не готово. Доведи до полного результата.

## 4. Работа с существующим кодом

Переиспользуй текущие компоненты и сущности.

Особенно важно:
- не создавать второй независимый 9-step compliance UI;
- не дублировать массивы demo-data рядом со старыми;
- не оставлять старые и новые элементы одновременно;
- не плодить новые маршруты, если можно переиспользовать существующие.

## 5. Документы и визуалы

Mock-документы:
- должны быть безопасными;
- только вымышленные данные;
- watermark `DEMO`;
- ретро-бюрократический пиксельный стиль можно использовать как tribute;
- не копировать игровые ассеты, имена стран/персонажей/логотипы.

Постеры:
- вертикальные;
- разные;
- похожие на реальные event-постеры;
- не одинаковый шаблон с разными цветами.

## 6. Проверка

После изменений обязательно:

- `npm run build`;
- smoke-check live/local;
- проверить, что `/main`, `/proto`, `/organizer`, `/admin`, `/channel`, `/demo` открываются;
- проверить, что ключевые flow живы;
- проверить, что GitHub Pages deploy успешен.
