## 3. Единый интерфейс заявки мероприятия

### 3.1. Проблема

Сейчас одна сущность «заявка мероприятия» отображается по-разному:
- в кабинете организатора;
- в compliance-форме;
- в Центре Управления.

Это ломает восприятие demo-flow.

### 3.2. Ожидаемая модель

Заявка мероприятия должна иметь один канонический интерфейс: 9 этапов compliance.

Разница только в режиме отображения:

#### Организатор
- видит список своих заявок;
- кликает по строке заявки;
- открывает 9-step compliance view этой заявки;
- поданные / одобренные заявки в основном read-only.

#### Центр Управления
- видит список заявок мероприятий;
- кликает по строке заявки;
- открывает тот же 9-step compliance view;
- дополнительно видит admin-действия: комментарий, одобрить, вернуть на доработку, отклонить.

### 3.3. Запрет

Нельзя создавать второй независимый интерфейс заявки.

Нужно переиспользовать существующий 9-step compliance UI и добавить режимы отображения, если это нужно технически.

## 8. Этап 5: назначение тарифов на схеме

### 8.1. Проблема

Сейчас тарифы назначены вертикальными полосами по колонкам.

Так в реальной билетной индустрии не делают.

### 8.2. Ожидаемая логика

Тарифы должны назначаться горизонтальными зонами по удалению от сцены:

- ближние ряды — более высокий тариф;
- средние ряды — стандартный тариф;
- дальние ряды / балкон / галерея — более дешёвый тариф.

### 8.3. Запрет

Не должно быть чередования тарифов по вертикальным колонкам.

Не должно быть шахматной/полосатой вертикальной раскраски.

## 9. Центр Управления: региональные режимы и данные

### 9.1. Режимы

Должны быть три режима:
1. Республиканский уровень / Супер-админ.
2. Региональный сотрудник: Могилёвская область.
3. Региональный сотрудник: Гродненская область.

### 9.2. Demo-data регионов

В Могилёвской и Гродненской областях должны быть реальные demo-данные.

Минимум:
- 2 заявки мероприятия для Могилёвской области;
- 2 заявки мероприятия для Гродненской области;
- площадки для этих регионов;
- события;
- решения;
- KPI-данные.

Не должно быть пустого режима, где пользователь выбрал регион и почти ничего не видит.

### 9.3. KPI по регионам

При переключении режима пересчитываются все регионально-зависимые KPI:

- новые заявки;
- на проверке;
- активные события;
- билеты выпущено;
- новые KPI, которые заменят «Нарушения» и «Подозрительные».

Сейчас проблема: меняются только часть цифр. Нужно исправить.

## 11. SeatMap constructor: шаблоны посадки

### 11.1. Проблема

Текущие шаблоны выглядят плохо:
- театр и амфитеатр похожи;
- цирк кривой;
- сектора налезают друг на друга;
- трибуны налезают на игровую зону;
- ледовая арена похожа на дворец спорта;
- малая арена похожа на цирк.

### 11.2. Ожидаемый результат

Шаблоны должны быть аккуратными, разными и demo-ready.

Это не production-конструктор, но визуально должно быть не стыдно показывать.

### 11.3. Шесть шаблонов

1. Театр  
   Сцена сверху, партер, балкон, ложи. Не копия амфитеатра.

2. Амфитеатр  
   Полукруглая / дуговая посадка вокруг сцены. Не копия театра.

3. Цирк  
   Круглая арена в центре, посадочные сектора вокруг. Без наложений.

4. Малая спортивная арена  
   Игровая зона в центре, трибуны вокруг. Без наложений, не копия цирка.

5. Ледовая арена  
   Лёд в центре, трибуны вокруг / по длинным сторонам. Не копия дворца спорта.

6. Дворец спорта / концертная арена  
   Концертная сцена или арена, сектора, VIP/ложи. Отличается от ледовой арены.

### 11.4. Запреты

Нельзя:
- оставлять наложения блоков;
- делать одинаковые шаблоны с разными названиями;
- добавлять open-air как seatmap-шаблон.

- концерт — сцена, свет, музыкант, хор, инструмент;
- театр — занавес, маска, силуэт актёра;
- цирк — арена, шатёр, прожектор;
- фестиваль — open-air, толпа, сцена, флаги;
- выставка — рамки, зал, арт-объект;
- классическая музыка — оркестр, скрипка, ноты, дирижёр.
