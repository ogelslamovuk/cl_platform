# Demo polish: data consistency, B2C, documents, regions, SeatMap templates

## Цель

Довести текущий live-прототип `cl_platform` после большого UI/UX-прохода до цельного demo-grade состояния.

Это полировочная итерация. Нельзя ломать существующий flow. Нужно исправить выявленные несоответствия, привести mock-data и UI к единой сквозной логике и довести результат до GitHub Pages.

## Контекст

Проект — демонстрационный прототип государственной платформы управления рынком культурных и массовых мероприятий.

Главный demo-flow, который нельзя ломать:

```text
Кандидат в организаторы -> Организатор -> Заявка на мероприятие -> Центр Управления -> Одобренное событие -> Реселлеры / розница / билеты
```

DTCM/Dubai — только продуктовый референс. Нельзя вводить Dubai/DTCM/NEN термины или отдельные сущности.

## Что нужно исправить

### 1. Mock-data consistency

Нужно:

- убрать мероприятия на 3000–3500 мест;
- оставить максимум 1–2 open-air/capacity-only события, каждое до 1500 мест;
- open-air всегда без схемы мест;
- события со схемой — максимум 500 мест;
- если событие имеет seatmap, его схема должна быть связана с площадкой из реестра площадок;
- если событие без схемы, нигде в UI не должна открываться seatmap;
- `kvitki.by` заменить на `bezkassira` / «Безкассира» во всех видимых местах и demo-data;
- не добавлять новые бессмысленные операции/флаги контроля;
- сохранить уже работающие флоу после PR #88.

### 2. Три режима Центра Управления

Нужно довести до понятной demo-модели три типа пользователей Центра Управления:

1. Республиканский уровень / супер-админ — видит всю страну.
2. Региональный сотрудник: Могилёвская область — видит только Могилёвскую область.
3. Региональный сотрудник: Гродненская область — видит только Гродненскую область.

У Могилёвской и Гродненской областей должны быть реальные mock-данные:

- площадки;
- заявки мероприятий;
- события;
- решения в журнале решений.

Регион заявки определяется по месту проведения / площадке, а не по юридическому адресу организатора.

### 3. Заявки мероприятий: фильтр по области

В разделе «Заявки мероприятий» добавить фильтр по области.

Требования:

- для республиканского уровня фильтр показывает все области;
- для регионального сотрудника список ограничен его областью;
- фильтр реально фильтрует данные;
- пустой региональный экран недопустим: у Могилёва и Гродно должны быть демонстрационные заявки.

### 4. Визуальные mock-документы

Документы в заявках сейчас открываются как бездушные текстовые popup. Нужно заменить/обогатить preview так, чтобы пользователь видел визуальный документ.

Где должно работать:

- заявка организатора: документы руководителя/сотрудников/работников, участвующих в организации;
- заявка мероприятия: программа мероприятия;
- договор/право использования площадки;
- документы/паспорта артистов и иностранных участников.

Стиль документов:

- ретро-пиксельная бюрократическая эстетика, tribute к погранично-документальным играм вроде `Papers, Please`;
- документы должны быть полностью самодельными через CSS/SVG/JSX;
- не использовать скриншоты, игровые ассеты, чужие изображения, логотипы, персонажей;
- не использовать реальные персональные данные;
- использовать вымышленные лица, страны, номера, даты;
- добавить watermark `DEMO`;
- документ должен визуально считываться как паспорт / виза / разрешение / договор / программа.

Минимальные типы визуальных mock-документов:

- `mock passport / ID card`;
- `mock entry visa / permit`;
- `mock venue contract`;
- `mock event program`;
- `mock participation confirmation`.

### 5. Compliance tooltips

Проверить все tooltips этапов compliance-заявки, а не только этап 5.

Нельзя оставлять generic-text вроде:

> Этап помогает сохранить черновик и вернуться к заявке позже, жёсткого порядка заполнения нет.

Все tooltips этапов должны быть смысловыми и описывать конкретный этап.

Итоговые тексты по этапам:

1. Основные сведения — `Укажите название, формат, категорию, возрастной рейтинг и базовые параметры мероприятия.`
2. Программа и участники — `Опишите программу, исполнителей, гражданство участников и приложите документы физических лиц.`
3. Показы / даты проведения — `Укажите даты, время проведения, дату начала реализации билетов и отдельные показы мероприятия.`
4. Площадка, зал и вместимость — `Выберите площадку, подтвердите право её использования, укажите вместимость и наличие схемы мест.`
5. Билеты и тарифы — `Задайте тарифы и базовые цены мероприятия. Назначение тарифов на конкретные места выполняется на схеме зала.`
6. Каналы продаж — `Выберите каналы и операторов, через которых будут продаваться билеты после одобрения события.`
7. Проверки и подтверждения — `Проверьте готовность документов, участников, программы, площадки и ведомственных mock-подтверждений.`
8. Пошлины и платежи — `Проверьте расчёт пошлины, основание начисления, статус оплаты или освобождение от оплаты.`
9. Проверка и подача — `Финально проверьте заявку, вернитесь к незаполненным этапам или подайте её в Центр Управления.`

Шаблоны:
