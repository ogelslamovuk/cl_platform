# Ожидаемый результат

Эта задача считается выполненной только при закрытии всех пунктов ниже. Частичное выполнение не принимается.

## 1. Общая цель

Прототип должен выглядеть как цельная бизнесовая платформа для управления рынком культурных и массовых мероприятий.

Сохраняется главный demo-flow:

`Кандидат в организаторы → Организатор → Заявка на мероприятие → Центр Управления → Одобренное событие → Операторы / розница / билеты`.

Все изменения должны быть demo-first: убедительно, кликабельно, понятно, но без production backend.

## 2. Кабинет организатора: шапка текущей заявки

На странице создания / редактирования заявки организатора должна появиться карточка текущего мероприятия.

Размещение: под вводным текстом страницы заявки, над плитками этапов.

Карточка показывает:

- название мероприятия;
- дату / время;
- площадку;
- город;
- область;
- статус заявки;
- короткий display ID заявки или события.

Карточка появляется после заполнения основных сведений или после заполнения demo-данными.

Карточка не должна ломать текущий stepper / плитки этапов.

## 3. Кабинет организатора: кнопка подачи заявки

Кнопка `Подать на рассмотрение` должна быть только на финальном этапе проверки и подачи.

На этапах 1–8 её быть не должно.

На этапах 1–8 допускаются только действия уровня:

- сохранить;
- сохранить и продолжить позже;
- далее / назад;
- локальные действия этапа.

Формулировку `обязательный минимум заполнен` убрать полностью.

Новая логика текста:

- заявка готова к подаче;
- заявка не готова к подаче, ниже перечислены конкретные незаполненные блоки.

## 4. Этап 2 заявки: программа и участники

Этап должен показывать две разные зоны:

1. `Программа мероприятия`.
2. `Участники и сопровождающие лица`.

### 4.1. Программа мероприятия

Блок программы показывает:

- описание программы;
- жанр / формат;
- возрастную категорию;
- краткое описание публичного исполнения;
- статус материалов программы.

### 4.2. Участники

Блок участников должен показывать структуру, а не один текстовый mock.

Типы записей:

- коллектив / группа;
- физическое лицо внутри коллектива;
- индивидуальный исполнитель;
- сопровождающая группа / технический персонал.

Для каждого физического лица показывать:

- ФИО;
- гражданство;
- роль;
- тип участия: исполнитель / сопровождающий / технический персонал;
- статус документа личности;
- статус mock-проверки.

Коллектив не должен отображаться как один человек.

Пример правильной логики:

- коллектив `Ансамбль "Спадчына"`;
- внутри 5 физических лиц;
- у каждого ФИО, гражданство, роль, документ, статус проверки.

### 4.3. Mock-паспорта / документы участников

Обязательно визуально показать, что документы участников прикрепляются.

Реальный upload backend не нужен.

Требуемый UI:

- карточка документа;
- иконка или миниатюра;
- название файла;
- статус `Приложен`;
- кнопка `Просмотреть`;
- при клике открывается modal / popup с mock-просмотром документа.

Документы должны быть безопасными mock-документами без реальных персональных данных.

Нельзя использовать настоящие паспорта, реальные ФИО реальных людей, реальные номера документов.
