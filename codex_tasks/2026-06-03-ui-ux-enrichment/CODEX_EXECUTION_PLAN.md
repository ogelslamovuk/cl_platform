# CODEX_EXECUTION_PLAN.md

# Автономный план реализации

Codex выполняет фазы подряд в одном запуске. Промежуточные approval у пользователя не запрашивать.

После каждой фазы выполнить локальный self-check этой фазы. После последней фазы выполнить общий build, smoke-check и live-delivery.

## Phase 0. Project inspection

Цель: понять текущую структуру проекта и минимальный набор файлов.

Действия:

1. Прочитать `CL_PLATFORM_CONTEXT.md` и `SKILLS.md`.
2. Найти текущие компоненты / страницы для:
   - Organizer dashboard;
   - Organizer event application flow;
   - Admin dashboard / Admin center;
   - Admin menu;
   - organizer applications;
   - event applications;
   - operators;
   - venue registry;
   - event registry;
   - tickets;
   - operations;
   - control;
   - decision log;
   - SeatMap / SeatMapModal / constructor;
   - B2C demo page;
   - shared mock-data / store / AppState.
3. Составить внутренний список файлов для изменения.
4. Не делать broad refactor.

Self-check:

- найден текущий источник mock-data;
- найден текущий источник маршрутов / views;
- найден существующий seatmap implementation;
- найден B2C demo route;
- найден admin menu.

## Phase 1. Organizer event application enrichment

Цель: довести заявку организатора до убедительного case-file UI.

Сделать:

1. Карточка текущего мероприятия вверху заявки.
2. Убрать `Подать на рассмотрение` с этапов 1–8.
3. Оставить финальную подачу только на этапе 9.
4. Убрать текст `обязательный минимум заполнен`.
5. Этап 2:
   - программа;
   - участники;
   - коллективы;
   - физические лица;
   - сопровождающие;
   - mock-документы / паспорта;
   - preview modal.
6. Этап 4:
   - договор с площадкой как document card;
   - preview modal;
   - upload-zone state when missing.
7. Этап 5:
   - заменить tooltip по тарифам.
8. Этап 7:
   - mock-проверки по лицам, иностранцам, программе, месту проведения.
9. Этап 8:
   - BYN сверху;
   - БВ снизу;
   - расчёт без выдуманных сборов;
   - понятное освобождение от пошлины;
   - явная подпись даты начала реализации билетов.
10. Этап 9:
   - кликабельные плитки этапов;
   - список недостающих блоков или `Заявка готова к подаче`.

Self-check:

- `Подать на рассмотрение` не встречается на этапах 1–8;
- mock-документы участников открываются;
- договор с площадкой открывается;
- BYN визуально основной показатель;
- этапы финального review кликабельны;
- текущий submit-flow заявки не сломан.

## Phase 4. Admin applications: organizers, events, operator applications

Цель: улучшить заявки и вынести заявки операторов в правильный раздел.

Сделать:

1. Заявки организаторов:
   - добавить фильтры;
   - default: новые / требующие рассмотрения;
   - улучшить читабельность.
2. Заявки мероприятий:
   - добавить больше данных в карточку / строку;
   - показать статусы документов, пошлины, проверок;
   - сделать документы кликабельными;
   - preview договора с площадкой.
3. Заявки операторов:
   - убрать из раздела `Операторы`;
   - перенести в `Регулятор → Заявки операторов`.

Self-check:

- заявки операторов не дублируются в двух местах;
- документы мероприятия открываются;
- фильтры работают;
- существующие approve/reject действия не сломаны.

## Phase 5. Regional model and registries

Цель: добавить область как demo-уровень прав и фильтрации.

Сделать:

1. Добавить / нормализовать область и город в mock-data для:
   - площадок;
   - мероприятий;
   - заявок мероприятий;
   - организаторов;
   - контрольных событий;
   - решений журнала.
2. Добавить demo-пользователей / режимы:
   - республиканский / супер-админ;
   - региональный сотрудник Могилёвской области.
3. Региональный сотрудник видит только данные своей области.
4. Фильтры по области добавить в:
   - реестр площадок;
   - реестр мероприятий;
   - реестр организаторов;
   - заявки мероприятий;
   - журнал решений;
   - отчёты с региональными данными.
5. Регион определяется по месту проведения / площадке, не по юридическому адресу организатора.

Self-check:

- супер-админ видит все регионы;
- региональный сотрудник видит только Могилёвскую область;
- фильтр по области работает;
- текущий admin-flow не сломан.

## Phase 6. Venue registry and event registry cleanup

Цель: сделать реестры компактными, логичными и без мусорных demo-кейсов.

Сделать:

1. Реестр площадок:
   - убрать кнопку `Заполнить mock-данными`;
   - сохранить географию;
   - для регионального сотрудника показывать только его область.
2. Реестр мероприятий:
   - добавить фильтры;
   - добавить область/город;
   - default sort: события с `Опубликовать` сверху;
   - крупные события не открывают seatmap;
   - крупные события показывают `Без схемы` / `Параметры входа`.
3. Demo-data:
   - seatmap-события не больше 500 мест;
   - open-air/capacity-only события могут быть больше 500 мест, но без seatmap.

Self-check:

- кнопка mock-fill из реестра площадок отсутствует;
- неопубликованные мероприятия сверху;
- крупное событие не открывает SeatMapModal как зал;
- события с местами продолжают открывать seatmap.

## Phase 7. Operators, tickets, operations, control, decision log

Цель: убрать дубли и бессмысленные mock-ситуации.

Сделать:

1. Операторы:
   - фильтры / сортировка по колонкам;
   - заявки операторов не здесь, а в `Регулятор → Заявки операторов`.
2. Билеты и операции:
   - разделы оставить отдельными;
   - удалить невозможную операцию `Продажа — отказ — нет доступных билетов`;
   - заменить на реалистичные события.
3. Контроль:
   - удалить бессмысленные нарушения до публикации;
   - не использовать `попытка публикации без оплаты пошлины`;
   - добавить реалистичные post-publication / operational риски.
4. Журнал решений:
   - добавить фильтры;
   - сделать читабельным.

Self-check:

- невозможная операция отказа продажи отсутствует;
- контрольные события имеют объект, регион, основание, действие;
- журнал решений фильтруется;
- билеты и операции не объединены.

Цель: добавить visual wow в существующий конструктор залов, не создавая второй конструктор.

   - Амфитеатр;
   - Цирк;
   - Малая спортивная арена;
   - Ледовая арена;
   - Дворец спорта / концертная арена.
2. Не добавлять open-air как seatmap template.
3. Разместить шаблоны в нижней левой свободной зоне конструктора.
4. Сделать их пиктограммами-карточками.
5. Клик заменяет текущую схему и обновляет счётчик мест.
6. Ограничить сложные схемы demo-пределом до 500 мест.

Self-check:

- используется существующий SeatMap / constructor;
- нет второго независимого реестра площадок;
- open-air не является seatmap template;
- 6 шаблонов работают;
- схема не обрезается контейнером.
