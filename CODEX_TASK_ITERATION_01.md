# CODEX_TASK_ITERATION_01.md

## Задача

Реализовать итерацию 1: превратить заявку организатора на мероприятие в многоэтапный wizard.

## Цель

Организатор заполняет заявку по этапам, сохраняет, продолжает позже, заполняет демо-данными, подаёт, получает возврат на доработку и снова продолжает. Центр Управления видит только поданные заявки, не черновики.

## Сначала найти минимальные файлы

Найти минимальный набор файлов для:

- страницы заявки организатора на мероприятие;
- admin/control center просмотра поданных заявок;
- shared state/demo data этих экранов;
- существующих tooltip/help components.

Менять только минимально нужные файлы.

## Этапы wizard — строго в этом порядке

1. Основные сведения
2. Программа и участники
3. Показы / даты проведения
4. Площадка, зал и вместимость
5. Билеты и тарифы
6. Каналы продаж
7. Проверки и подтверждения
8. Пошлины и платежи
9. Проверка и подача

## Требования к организатору

- Видимый stepper/wizard вместо линейной формы.
- Свободный переход между этапами, без жёсткого порядка.
- Прогресс и статусы этапов только для организатора.
- Статусы: Не заполнено / Черновик / Заполнено / Требует внимания.
- Кнопки: Сохранить; Сохранить и продолжить позже; Заполнить демо-данными; Далее; Подать на рассмотрение.
- Сохранение работает через текущий state/localStorage flow.
- После подачи заявка появляется в Центре Управления.
- Черновики не появляются в Центре Управления.

## Этап 1: Основные сведения

Добавить demo-каталог типа мероприятия на 3 уровня. Не строить большой реальный классификатор.

Минимальное дерево:

- Культура -> Концерт -> Эстрадный концерт
- Культура -> Театр -> Драматический спектакль
- Спорт -> Индивидуальный -> Стрельба из лука
- Спорт -> Индивидуальный -> Бег
- Спорт -> Командный -> Футбол
- Бизнес -> Конференция -> Отраслевая конференция

Показать выбранный путь как бейдж/строку. Сохранять в текущие данные заявки или совместимое поле, не создавать параллельную сущность.

## Этап 2: Программа и участники

- программа/описание;
- список участников/исполнителей;
- ручной лимит максимум 10 участников;
- если добавляют больше 10 — русское уведомление и участник не добавляется;
- mock-документы участников;
- для иностранных участников — mock-контекст документов/проверок;
- demo-fill не создаёт больше 10 видимых участников;
- реальные паспортные данные не использовать.

## Этап 3: Показы / даты проведения

Показать несколько дат/слотов, совместимых с текущим flow. Не создавать production performance engine.

## Этап 4: Площадка, зал и вместимость

Выбор площадки только из существующего реестра.

Жёстко:

- есть в реестре — можно выбрать;
- нет в реестре — нельзя добавить из wizard;
- не создавать площадку из заявки;
- не добавлять flow заведения площадки.

Добавить обязательную mock-строку документа: `Договор с площадкой`. Показать статус документа: приложен / требуется / образец. Генератор договора не делать.

## Этап 5: Билеты и тарифы

Использовать только текущий конструктор схемы зала / тарифов.

Запрещено:

- новый конструктор схемы зала;
- Excel-like import/export тарифов;
- DTCM-style ручная тарификация через Excel;
- замена текущего конструктора;
- дублирование текущего механизма.

## Этап 6: Каналы продаж

Показать каналы продаж/реселлеров через существующие структуры. Глубоко reseller/retail logic не перестраивать.

## Этап 7: Проверки и подтверждения

Добавить mock-блок межведомственных проверок:

- МВД — проверка документов участников;
- Департамент по гражданству и миграции — проверка иностранных участников;
- Минкульт — проверка организатора и основания проведения;
- Исполком — уведомление по площадке и проведению мероприятия.

Mock-действия:

- Сформировать пакет
- Отправить на проверку
- Обновить статус

Статусы:

- Не отправлено
- Отправлено
- В обработке
- Проверено
- Требует уточнения

Реальные интеграции не делать.

## Этап 8: Пошлины и платежи

Оставить лёгким: показать текущий статус пошлины/оплаты и вложения платежей, если уже есть. Баланс кабинета / financial account не делать — это итерация 2.

## Этап 9: Проверка и подача

Показать summary всех этапов. Показать незаполненные обязательные разделы. Разрешить подачу только при заполненном обязательном минимуме.

## Центр Управления

Центр Управления не видит черновики.

Для поданных заявок админ может:

- открыть детали;
- видеть заполненные разделы;
- видеть категорию мероприятия;
- видеть участников/документы;
- видеть площадку и статус договора;
- видеть межведомственные проверки;
- видеть статус пошлины/оплаты;
- оставить одно общее замечание;
- вернуть на доработку;
- одобрить;
- отклонить.

В этой итерации только одно общее замечание. Не делать комментарии к каждому полю. Организатор должен явно видеть замечание после возврата.

## Tooltips/help

Все новые поля, блоки и неочевидные действия должны иметь HelpTooltip или существующий help-паттерн.

Жёстко:

- tooltip text только русский;
- не использовать HTML title как основной tooltip;
- не помещать HelpTooltip внутрь button;
- не ставить tooltips на table headers/plain values без необходимости.

## Дизайн

Текущий стиль сохранить. Продукт не редизайнить. Новые библиотеки не добавлять. Branding не менять. Английский UI-текст не добавлять. Меню/навигацию не менять без крайней необходимости.

## Запрещено

- DTCM/NEN/Dubai сущности;
- новая сущность заявки параллельно EventComplianceApplicationRecord;
- создание площадки из wizard;
- новый конструктор схемы зала/тарифов;
- Excel workflow по местам/тарифам;
- реальные госинтеграции;
- реальный payment gateway;
- глубокая перестройка реселлеров/розницы;
- поломка цепочки кандидат -> организатор -> заявка -> админ -> событие;
- unrelated pages;
- backend;
- зависимости без жёсткой необходимости.

## Demo scenario после деплоя

1. Открыть страницу заявки организатора.
2. Нажать Заполнить демо-данными.
3. Проверить 9 этапов.
4. Проверить свободный переход.
5. Проверить 3-уровневый каталог и выбранный путь.
6. Проверить участников/документы и лимит 10.
7. Проверить выбор площадки из реестра и договор с площадкой.
8. Проверить, что билеты/тарифы используют текущий constructor/flow.
9. Проверить mock межведомственные проверки.
10. Сохранить и продолжить позже.
11. Вернуться и проверить сохранение данных.
12. Подать заявку.
13. Открыть Центр Управления.
14. Проверить, что поданная заявка видна.
15. Проверить, что черновики не видны.
16. Добавить общее замечание и вернуть на доработку.
17. Открыть кабинет организатора и проверить, что замечание явно видно.
18. Одобрить заявку и проверить, что текущий post-approval flow не сломан.
