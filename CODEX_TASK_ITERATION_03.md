# CODEX_TASK_ITERATION_03.md

## Задача

Реализовать Iteration 3: удобный инструмент расценивания мест в зрительном зале через текущий конструктор схемы зала и тарифов.

## Главная цель

Организатор должен уметь удобно собрать билетную конфигурацию зала:

- назначить базовые цены;
- массово применить тариф к сектору/ряду/группе мест;
- точечно изменить тариф на отдельных местах;
- показать один демонстрационный льготный тариф;
- сохранить тарифные назначения;
- увидеть понятный summary по местам, тарифам, исключениям и расчётной максимальной выручке.

Центр Управления должен видеть результат только в read-only.

## Что НЕ является целью

Эта итерация НЕ про расписание и НЕ про несколько показов под одно мероприятие.

Запрещено:

- добавлять список показов;
- создавать sessions / performances / showtimes;
- привязывать тарифы к нескольким показам;
- менять этап “Показы / даты проведения”;
- строить расписание события.

Эта итерация только про удобное расценивание мест в зале.

Менять только минимально нужные файлы.

## 1. Использовать только текущий конструктор

Нужно усилить существующий конструктор схемы зала и тарифов.

Нельзя создавать новый конструктор.

Запрещено:

- новый альтернативный SeatMapConstructor;
- новый отдельный модуль конструктора;
- Excel-like import/export;
- DTCM-style manual tariff table;
- отдельный route для нового конструктора;
- замена существующего конструктора новым;
- новая параллельная сущность SeatMapEngine/TariffEngine, если можно расширить текущие данные.

Если текущий конструктор проще, чем ожидается, не останавливайся и не создавай новый. Реализуй минимальную полную версию поверх существующих seat objects и текущего data flow.

## 2. Массовое назначение тарифа

Добавить возможность массового назначения тарифа на группу мест.

Приоритетный сценарий:

- назначить тариф на сектор / зону / ряд / группу мест, если такие группы уже есть в текущей модели.

Fallback-сценарий, если в текущей модели нет секторов/групп/рядов:

- не создавать новую модель секторов;
- реализовать bulk assignment через выделение нескольких существующих мест на текущей схеме;
- пользователь выбирает несколько мест, выбирает тариф и применяет его к выбранным местам.

Обязательный UI-сценарий:

1. Выбрать тариф.
2. Выбрать сектор/ряд/группу/несколько мест.
3. Нажать действие применения.
4. Увидеть результат на схеме и в summary.

Примеры русских UI-текстов:

- “Назначить тариф группе мест”
- “Выберите тариф”
- “Применить к сектору”
- “Применить к выбранным местам”
- “Места с этим тарифом”

Все тексты только на русском.

## 3. Индивидуальные исключения по местам

Добавить возможность точечно изменить тариф отдельного места.

Обязательный сценарий:

1. На группу мест назначен тариф, например “Стандарт — 100 BYN”.
2. Пользователь выбирает отдельное место внутри этой группы.
3. Назначает ему другой тариф, например “Премиум — 150 BYN”.
4. Это место отображается как индивидуальное исключение.

Нужно показать:

- индивидуальный тариф места;
- что это исключение относительно группового тарифа;
- количество индивидуальных исключений в summary.

Желательно добавить действие:

- “Вернуть тариф группы”
- или “Сбросить индивидуальный тариф”

Если это слишком сложно в текущей модели, достаточно возможности заменить индивидуальный тариф другим тарифом и явно считать его исключением.

## 4. Тарифы

Конструктор должен поддерживать демонстрационный набор тарифов.

Минимум:

- Стандарт — 100 BYN
- Премиум — 150 BYN
- VIP — 200 BYN
- Льготный (пример) — 50 BYN

Льготный тариф нужен только для демонстрации того, что платформа учитывает социальные/льготные категории билетов.

Важно:

- не строить справочник льгот;
- не проверять право на льготу;
- не добавлять удостоверения льготника;
- не добавлять документы для льгот;
- не делать юридическую систему льгот;
- не перечислять много льготных категорий.

Достаточно одного видимого примера:

- “Льготный (пример) — 50 BYN”
- или “Льготный (пример: детский/социальный) — 50 BYN”

## 5. Применение льготного тарифа

Льготный тариф должен работать как обычный тариф в конструкторе:

- его можно применить к группе мест;
- его можно применить к отдельному месту;
- он может быть индивидуальным исключением внутри сектора/группы.

В summary отдельно показать:

- “Льготные места”
- количество мест с льготным тарифом.

Не делать проверку права на льготу.

## 6. Summary в заявке

На этапе “Билеты и тарифы” показать summary:

- всего мест в схеме;
- мест с назначенным тарифом;
- мест без тарифа;
- количество мест по каждому тарифу;
- количество льготных мест;
- количество индивидуальных исключений;
- ориентировочная максимальная выручка по назначенным тарифам.

Это демонстрационный расчёт, не бухгалтерский отчёт.

## 7. Сохранение и отображение после подачи

Назначенные тарифы, индивидуальные исключения и льготные места должны сохраняться в текущем state/localStorage flow.

Обязательные требования:

- после “Сохранить” данные не теряются;
- после reload/возврата к заявке данные не теряются;
- после подачи заявки тарифная конфигурация отображается в Центре Управления;
- read-only summary в Центре Управления показывает те же тарифы/исключения/льготные места, которые задал организатор.

Не делать отдельное production-хранилище.

## 8. Центр Управления

В просмотре поданной заявки Центр Управления должен видеть read-only summary билетной конфигурации:

- выбранная площадка/зал;
- вместимость;
- тарифы;
- количество мест по тарифам;
- льготные места;
- индивидуальные исключения;
- максимальная расчётная выручка.

Админ не редактирует схему и тарифы.

Админ только проверяет, что организатор собрал билетную конфигурацию.

## 9. Связь с текущим flow

Нельзя ломать цепочку:

```text
Кандидат в организаторы -> Организатор -> Заявка -> Центр Управления -> Одобренное событие -> Реселлеры / розница / билеты
```

После approval текущий flow должен работать как раньше.

Если текущий event/ticket flow уже использует тарифы из заявки, сохранить совместимость.

Если не использует — не строить новый production bridge в этой итерации. Достаточно не ломать текущие данные и показать summary.

## 10. Tooltip/help

Все новые поля, блоки и неочевидные действия должны иметь HelpTooltip или существующий help-паттерн проекта.

Жёстко:

- использовать существующий HelpTooltip/help component, если он есть;
- не использовать HTML title как основной tooltip;
- не помещать HelpTooltip внутрь button;
- tooltip text только на русском;
- не ставить tooltips на plain values/table headers без необходимости.

## 11. UI/design/language

- весь новый UI-текст только на русском;
- не менять общий стиль;
- не редизайнить страницу заявки;
- не менять branding;
- не добавлять новые UI libraries;
- не добавлять английские тексты;
- не менять navigation/menu без необходимости.

## 12. Запрещённый scope

Нельзя:

- делать показы / sessions / performances;
- создавать новый конструктор схемы зала;
- создавать новую модель секторов, если её нет в текущей схеме;
- создавать Excel import/export;
- строить production ticketing engine;
- строить справочник льгот;
- проверять право на льготу;
- добавлять документы/удостоверения льготника;
- трогать payment/balance flow из Iteration 2 без необходимости;
- ломать wizard из Iteration 1;
- глубоко перестраивать реселлеров/розницу;
- создавать backend;
- добавлять dependencies без жёсткой необходимости;
- менять unrelated pages.

## Demo scenario после деплоя

Codex должен проверить на deployed site:

1. Открыть заявку организатора.
2. Перейти на этап “Билеты и тарифы”.
3. Убедиться, что используется существующий конструктор схемы, а не новый отдельный модуль.
4. Выбрать тариф “Стандарт”.
5. Массово применить тариф к сектору/ряду/группе/нескольким местам.
6. Убедиться, что схема и summary обновились.
7. Сохранить заявку.
8. Обновить страницу или вернуться к заявке.
9. Убедиться, что тарифные назначения сохранились.
10. Выбрать отдельное место внутри этой группы.
11. Назначить ему тариф “Премиум”.
12. Убедиться, что место отображается как индивидуальное исключение.
13. Выбрать тариф “Льготный (пример)” и применить его к одному месту.
14. Убедиться, что summary показывает льготное место.
15. Убедиться, что summary показывает количество исключений.
16. Убедиться, что summary показывает расчётную максимальную выручку.
17. Сохранить заявку.
18. Подать заявку.
19. Открыть Центр Управления.
20. Открыть поданную заявку.
21. Убедиться, что read-only summary билетной конфигурации виден.
22. Убедиться, что read-only summary показывает те же тарифы, исключения и льготные места.
23. Убедиться, что админ не может редактировать тарифы.
24. Убедиться, что Iteration 1 wizard не сломан.
25. Убедиться, что Iteration 2 payment approval gate не сломан.
26. Убедиться, что approval и post-approval flow работают как раньше.
