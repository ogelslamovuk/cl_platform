# CODEX_TASK_ITERATION_02.md

## Задача

Реализовать итерацию 2: Финансовый счёт / Баланс организатора / Fees & Payments.

## Цель

Сделать убедительный сквозной mock финансового слоя платформы.

Организатор видит баланс в dashboard, проваливается в финансовый счёт, mock-пополняет баланс, оплачивает обязательные fees по заявке с баланса.

Центр Управления видит статус оплаты по поданной заявке и не может одобрить заявку до оплаты обязательных fees.

## Важнейшая бизнес-логика

В этой итерации в продукте НЕ существует сущности “долг”.

Запрещено использовать в новом UI и новой логике понятия:

- долг;
- задолженность;
- просрочка;
- debtor;
- overdue.

Правильная модель:

1. Заявку можно подать без оплаты.
2. После подачи она видна в Центре Управления.
3. Неоплаченная заявка имеет статус оплаты “Ожидает оплаты”.
4. Админ может открыть и посмотреть неоплаченную заявку.
5. Одобрить неоплаченную заявку нельзя.
6. Approval становится доступен только после оплаты обязательных fees.
7. После оплаты статус становится “Оплачено”.

## 1. Dashboard организатора

В dashboard организатора найти карточку/показатель:

- “К оплате платформе”
- или ближайший аналогичный текущий показатель по оплате платформе.

Заменить на карточку:

- “Баланс”

Карточка должна показывать текущий баланс организатора.

Карточка “Баланс” должна быть кликабельной и вести в раздел “Финансовый счёт”.

Если в проекте уже есть подходящий способ перехода между разделами внутри кабинета организатора — использовать его.

Если отдельный route не нужен, допускается секция/экран внутри текущего organizer flow, но переход по клику с dashboard обязателен.

## 2. Финансовый счёт организатора

Создать/показать раздел “Финансовый счёт” в кабинете организатора.

Раздел должен содержать:

- текущий баланс;
- доступно к использованию;
- зарезервировано, только если это естественно ложится на текущий mock;
- mock-кнопку “Пополнить счёт”;
- историю операций;
- операции по заявкам;
- mock-квитанции/receipts/vouchers;
- возврат в dashboard организатора;
- связь с конкретной заявкой, если операция относится к заявке.

Все тексты на русском.

Не делать реальный payment gateway.

## 3. Mock-пополнение баланса

Кнопка “Пополнить счёт” должна делать mock-действие:

- увеличивает баланс на демонстрационную сумму;
- добавляет операцию в историю;
- показывает понятный русский статус/feedback.

Не подключать банк.
Не открывать внешний gateway.
Не добавлять реальные платежные интеграции.

## 4. Этап заявки “Пошлины и платежи”

В существующем wizard из итерации 1 усилить этап:

- “Пошлины и платежи”

Этап должен показывать:

- начисленные fees по заявке;
- сумму к оплате;
- текущий баланс;
- статус оплаты;
- кнопку “Оплатить с баланса”;
- кнопку “Сформировать квитанцию”;
- историю/след операции по этой заявке;
- понятное сообщение, если средств недостаточно.

Статусы оплаты использовать строго:

- Ожидает оплаты
- Недостаточно средств
- Оплачено
- Средства зарезервированы — только если удобно и не усложняет текущий flow

Не использовать “долг”, “задолженность”, “просрочено”.

## 5. Оплата с баланса

Кнопка “Оплатить с баланса” должна:

- проверить, хватает ли средств;
- если средств хватает — списать mock-сумму с баланса;
- добавить операцию в историю финансового счёта;
- обновить статус оплаты заявки на “Оплачено”;
- показать mock-квитанцию/receipt/voucher;
- сделать заявку доступной к approval в Центре Управления.

Если средств не хватает:

- баланс не менять;
- статус/feedback: “Недостаточно средств”;
- показать действие “Пополнить счёт” или понятную подсказку;
- не использовать слово “долг”.

## 6. Центр Управления

В Центре Управления для поданных заявок добавить/показать:

- статус оплаты;
- сумму fees;
- факт оплаты;
- квитанцию/receipt, если оплачено;
- визуальную подсветку заявок со статусом “Ожидает оплаты”;
- пояснение: “Одобрение доступно после оплаты обязательных пошлин.”

Поведение approval:

- неоплаченную заявку нельзя одобрить;
- кнопка approve должна быть disabled или действие должно быть заблокировано с понятным русским объяснением;
- после оплаты approval работает как раньше.

Админ не управляет балансом организатора.

В Центре Управления нет сущности “долг”.

## 7. Связь с текущим flow

2. Убедиться, что карточка “К оплате платформе” заменена на “Баланс”.

4. Убедиться, что открыт раздел “Финансовый счёт”.
5. Убедиться, что видны баланс, доступно, история операций, mock-квитанции.
6. Нажать “Пополнить счёт”.
